# mvp-sysbuild - 系统工具 MVP 开发框架

## 核心原则

> 1. **先问清楚，再动手。** 不猜测用户的真实意图，不假设技术方案。
> 2. **用户确认即锁定。** 任何经用户确认的决定（技术选型、功能范围、UI 设计等），AI 不可单方面更改；如需更改，必须再次向用户说明原因并征得确认。
> 3. **默认不运行代码。** 除非用户明确要求"运行/执行/测试代码"或"帮我找出错误并解决"，否则只写代码、不执行。
> 4. **运行/打包超时主动申请。** 若代码运行或打包超过合理时间无反馈，主动请求用户来执行。

## 代码执行规则

### 何时可以运行代码

| 场景 | 是否运行 | 说明 |
|------|----------|------|
| 用户明确说"运行"、"执行"、"测试" | ✅ 可以运行 | 按用户要求执行 |
| 用户说"帮我找出错误并解决" | ✅ 可以运行 | 为调试目的运行 |
| 用户只说"写代码"、"实现功能" | ❌ 不运行 | 只写不跑 |
| 用户问"这段代码有什么问题" | ❌ 不运行 | 静态分析，不执行 |

### 运行/打包超时处理

当代码运行或打包**超过合理时间**仍未返回结果时：

1. **立即终止等待**
2. **主动告知用户**："运行/打包耗时较长，建议由你来执行"
3. **提供确切命令**：给出完整的运行/打包命令，方便用户执行
4. **说明预期结果**：告诉用户执行后应该看到什么输出

**合理时间参考**：
- 代码运行（脚本/程序）：> 30 秒无输出 → 申请用户执行
- 项目构建（cargo build / npm run build）：> 2 分钟无输出 → 申请用户执行  
- 依赖安装（npm install / cargo build）：> 5 分钟无输出 → 申请用户执行

### 示例

✅ **正确做法**（用户未要求运行）：
```
用户：帮我写一个 RSS 解析函数
AI：好的，这是代码：[代码]
```

✅ **正确做法**（用户要求运行）：
```
用户：运行一下这段代码，看看有没有错误
AI：[执行代码，返回结果]
```

✅ **正确做法**（运行超时）：
```
AI：这段代码运行时间较长，建议你在本地执行：
命令：cargo run --release
预期：看到 "Server started on port 8787" 输出
```

❌ **错误做法**（用户未要求却运行）：
```
用户：帮我写一个 RSS 解析函数
AI：[直接运行代码]  ← 不应该！
```

## 用户确认规则（强制执行）

以下场景**必须**获得用户明确确认，且确认后不可单方面更改：

| 决策类型 | 示例 | 更改时需再次确认 |
|----------|------|------------------|
| 技术选型 | "使用 Tauri + React" | 必须说明原因，用户同意后才可切换 |
| 功能范围 | "支持环境变量+hosts管理" | 新增/删减功能都需确认 |
| 用户故事 | 6 个用户故事的验收标准 | 修改任何验收标准都需确认 |
| UI 设计 | "深色主题、左侧导航" | 更改风格/布局需确认 |
| 数据存储 | "SQLite 本地存储" | 换存储方案需确认 |

**AI 不得因为"我觉得更好"、"遇到技术困难"等原因自行更改已确认的决定。** 遇到困难时，应向用户说明情况并提出替代方案，由用户选择。

## 完整流程

```
Phase 1: 需求确认 ← 【必须先做】
  ├─ 确认工具形态（CLI / Web UI / 桌面 GUI / Android / iOS）
  ├─ 确认技术栈（从推荐方案中选，用户决定）
  ├─ 生成需求简报，征得用户确认 ✅
  └─ 确认结果记入项目上下文，后续变更需再次确认

Phase 2: 生成用户故事
  ├─ 基于确认的需求，拆解为可测试的用户故事
  ├─ 展示用户故事，征得用户确认 ✅
  └─ 确认后的故事如有修改，需再次确认

Phase 3: 执行开发
  ├─ 搭建项目骨架（Tauri / Next.js / CLI 等）
  ├─ 逐条实现用户故事
  ├─ 每个故事完成后即时演示结果
  └─ 遇到技术困难 → 暂停、说明、提方案、等确认

Phase 4: 收尾
  ├─ 告知工具路径和使用方法
  └─ 记录可扩展方向
```

## Phase 1: 需求确认

触发技能后，AI **不得立即写代码**，必须先与用户确认以下问题：

### 问题 1：目标平台 + 工具形态

> "你希望这个工具跑在什么平台上？是什么形式？"

| 平台 | 形态选项 | 说明 |
|------|----------|------|
| **Windows** | CLI / 桌面 GUI / Web UI | 最常见场景 |
| **macOS** | CLI / 桌面 GUI / Web UI | 需考虑 Apple Silicon |
| **Linux** | CLI / 桌面 GUI / Web UI | 桌面可选 GTK/Qt |
| **Android** | 原生 App / 跨平台 App | |
| **iOS** | 原生 App / 跨平台 App | |
| **Web** | SPA / SSR / 静态站 | 浏览器访问 |

> 可以多选，例如"Windows 桌面 + Web 管理界面"。

### 问题 2：技术栈（从推荐方案中选，用户决定）

根据平台和工具形态，提供 2-3 个推荐方案：

#### Windows 桌面 GUI 推荐方案

| 方案 | 技术栈 | 适合场景 | 特点 |
|------|--------|----------|------|
| **推荐** | Tauri 2.x + React + TypeScript | 需要本地系统集成的工具 | 轻量(8MB)、美观、无需安装运行时、系统权限 |
| **备选** | Electron + React + TypeScript | 需要复杂 UI / 多窗口 | 生态成熟、兼容性好、包体积大(150MB+) |
| **备选** | PyQt6 + Python | 内部工具、脚本胶水层 | Python 生态丰富、学习曲线平缓 |

#### Windows CLI 推荐方案

| 方案 | 技术栈 | 适合场景 |
|------|--------|----------|
| **推荐** | Go (单文件) | 需要系统权限的 CLI 工具 |
| **推荐** | Python + PyInstaller | 快速原型、需要调用脚本 |
| **备选** | Node.js (pkg) | TypeScript 项目复用 |

#### Web 应用推荐方案

| 方案 | 技术栈 | 适合场景 |
|------|--------|----------|
| **推荐** | Next.js + SQLite | 需要本地数据存储的 Web 工具 |
| **推荐** | Next.js + PostgreSQL | 需要多端访问的管理后台 |

**无论哪种方式，最终选型必须用户明确确认，确认后记入项目上下文。**

### 需求简报

收集完上述信息后，AI 生成需求简报并征得确认：

```
===== 需求简报 =====
工具名称：EnvHost Manager
核心功能：环境变量管理 + hosts 文件管理
目标平台：Windows
工具形态：桌面 GUI
技术方案：Tauri 2.x + React + TypeScript (前端) + Rust (后端)
推荐理由：轻量(8MB)、美观、无需安装运行时
===================
确认以上需求吗？确认后开始开发。
```

**如果用户跳过问题直接说"随便"/"你决定"，则 AI 自主决定并标注「默认方案」，再进入 Phase 2。**

## Phase 2: 生成用户故事

基于确认的需求，拆解为具体可测试的用户故事：

### 格式

```
作为 [用户角色]，
我想要 [做某件事]，
以便 [实现某个目标]。

验收标准：
- [可测试的检查点 1]
- [可测试的检查点 2]
```

### 展示方式

将所有用户故事一次性展示，请用户：
- 确认通过的
- 修改某个故事的内容
- 补充遗漏的故事
- 删掉不必要的故事

**确认后记入项目上下文，后续修改任何验收标准都需再次确认。**

## Phase 3: 执行开发

按确认后的用户故事逐条实现。

### Tauri 项目标准骨架

```
project/
├── src/                        # 前端 (React/Vue/Svelte)
│   ├── App.tsx
│   ├── App.css
│   ├── main.tsx
│   ├── api.ts                  # Tauri invoke 封装
│   ├── types.ts                # 前后端共享类型
│   └── components/
├── src-tauri/
│   ├── src/
│   │   ├── lib.rs              # 主库入口，app builder 配置
│   │   ├── main.rs             # bin 入口 (必须！)
│   │   ├── db.rs               # 数据库层 (可选)
│   │   └── commands.rs         # Tauri 命令 (可选)
│   ├── Cargo.toml              # 必须含 [[bin]] 段！
│   ├── tauri.conf.json
│   └── icons/                  # 必须有 icon.ico + icon.png
├── package.json
├── vite.config.ts
└── index.html
```

**⚠️ Tauri 项目常见遗漏（导致编译失败）：**
1. `src-tauri/src/main.rs` 缺失 → `cargo run` 报错 "a bin target must be available"
2. `Cargo.toml` 缺少 `[[bin]]` 段 → 同上
3. `src-tauri/icons/` 缺失 → `tauri build` 报错 "icon not found"
4. `tauri.conf.json` 的 `devtools` 在 Windows 下默认 false，开发时设为 true

### 已确认决策的变更流程

开发过程中如遇技术困难，需更改已确认的决定：

1. **暂停当前开发**
2. **向用户说明**：
   - 原决策是什么
   - 遇到什么问题
   - 建议的替代方案（可提供 2-3 个选择）
   - 每个方案的利弊
3. **等用户选择并确认**
4. **确认后才可继续**

### 技术规范

- 脚本输出用 **JSON 格式**，便于展示层解析
- Windows GUI 优先 **Tauri**（轻量美观）
- 错误信息对用户友好，不暴露技术细节
- 文件写入遵循 qclaw-text-file 技能规范（无 BOM UTF-8 / GBK 自适应）
- Windows 注册表操作：`KEY_READ | KEY_WRITE` 缺一不可（仅 KEY_WRITE 会导致无法读取原值）

## 常见问题速查

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| `cargo run` 报错 "no bin target" | Cargo.toml 只有 `[lib]` | 添加 `[[bin]]` 段 + 创建 `main.rs` |
| `icon not found` | 缺少 `src-tauri/icons/` | 从 envhost 项目复制图标或生成占位图 |
| `npm run tauri dev` 端口占用 | 旧进程未退出 | `Stop-Process -Id <pid>` 或改 `tauri.conf.json` 的 `devPort` |
| PowerShell 调用 Python 中文乱码 | 编码问题 | 用无 BOM UTF-8 写文件 + `curl.exe` / `python` 而非 `python -c` |
| Windows 注册表写入后原值类型丢失 | `RegSetValueEx` 未保留 `REG_EXPAND_SZ` | 读取时保存 `RegQueryValueEx` 返回的类型，写入时使用同一类型 |
| Node.js 版本冲突（env/Project 不一致） | PATH 中 Node 版本与项目要求不同 | 在项目 `package.json` 中用绝对路径，或创建 `scripts/dev.cmd` 用绝对路径调用 |
